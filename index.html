GSoC Update: July 24 2013

Jul 24th, 2013 | Comments

After working on the django models for Oppia we finally have a running heroku app at oh-oppia.herokuapp.com. I have also worked on a way to combine the oppia WSGI app and Openhatch WSGI app using Werzeug’s DispatcherMiddleware. I have also worked on the auth system, so that I can pull out the session from webapp2 request object and passing into django middlewares to get the authenticated user out of it.

The next thing I will be working on is the routing, which is currently broken after combining both WSGI apps. Also I’ll be trying to push all my changes to Oppia mainline.

That’s all for now.

GSoC Update: July 11 2013

Jul 11th, 2013 | Comments

Last week we made some significant changes to the way models are structured in Oppia. They are now divided into 3 files named models.py, services.py, domain.py. While models.py holds the platform dependent model code, the other 2 are completely independent of the platform. Hence we are now in a position to use GAE datastore models directly without using django-nonrel.

Other than that, I mostly worked on merging changes from upstream and fixing the merged code. I have also implemented a to_dict method which represents a model as a dictionary. Statistics model is somewhat ported for now. It still has some failing tests.

That’s it for now.

Cheers.

GSoC Update: June 29 2013

Jun 29th, 2013 | Comments

Last week I mostly worked on porting the models. Oppia uses LocalStructuredProperty available by GAE datastore a lot to store model instances locally inside other models. We are using a customized version of django-json-field to implement that in Django by overriding the methods which sets and gets the values from from the database inside JSONField. Basically we are converting the objects to a JSON string before storing in the database and then retrieving the object from that JSON string while fetching it again from the database.

Next week, I’ll merge the upstream changes. Some changes in exploration and state model has been done to simplify things. Ancestor Queries have been removed. So I’ll try to implement those changes in the Django models. Also I will add some serialization code which is needed to represent models as nested dictionaries.

That’s it for now.

Cheers.

Porting Oppia to Django

Apr 27th, 2013 | Comments

For those who are wondering what oppia is,

Oppia is a web-based tool for creating interactive online activities that enable students to learn by doing. Oppia aims to capture and generalize the one-on-one “conversations” that a student has with a teacher, so that other students can benefit from them.

Currently, oppia is written using webapp2 and runs on Google App Engine only. The goal is to port it to django, so that it can run on more number of platforms and hopefully be used for training missions at openhatch
